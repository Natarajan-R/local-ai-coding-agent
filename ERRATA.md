# Errata — *Building a Local AI Coding Agent*

Corrections to the published edition, found by running **every chapter's code as an end
user** — driving the agent the way you would, rather than calling its internals from a
test. Each one is already fixed in this repository, so if you are working from the code
here you have the corrected version. The Kindle edition is updated periodically.

Reproduce any of these yourself with **[`EXAMPLES.md`](EXAMPLES.md)**.

| # | Where | What the book says | The correction |
| --- | --- | --- | --- |
| 1 | **Preface** §2 (model table) | Describes `qwen2.5:7b` as "Qwen 2.5 **Coder** 7B" | It is **Qwen 2.5 7B Instruct**, not the Coder variant (`qwen2.5-coder:7b`). The repo's own evaluation is why: Instruct beat Coder **12/12 vs 10/12** on the edit-an-existing-file loop. |
| 2 | **Ch 15** — Sandbox Manager Facade | `"auto"` picks Docker whenever the daemon is reachable | That fails with a container-pull 404 when the **image was never built**. `"auto"` now also checks the image exists, and falls back to the local sandbox with a hint to run `ai-agent build-sandbox`. |
| 3 | **Ch 42** — The Dashboard Page | The reference UI is display-only | It is now a self-contained control surface: task input, Run button, and a live activity feed. You can drive a whole run from the browser. |
| 4 | **Ch 10** — Execution Loop, **Ch 20** — Secret Redaction | Secrets are "redacted before the model ever sees it" | Redaction was only applied to `run_command` output, so **`read_file` leaked secrets into the model's context and the console**. Every tool result is now scrubbed once at the boundary — which is what Ch 20 always claimed. |
| 5 | **Preface** §3, **Ch 4**, both appendices | `pip install python-lsp-server` | **pylsp alone reports no diagnostics** — it ships no linter plugins, so `get_diagnostics` (Ch 27–31) silently returns "No diagnostics reported." Install `python-lsp-server[pyflakes,pycodestyle]` instead. |
| 10 | **Ch 13** — Subprocess Sandbox, and the command guard (**Ch 17/18**) | Ch 13 says the local sandbox's isolation is "directory-scoped… and the guardrails from Part IV keep the model from asking to run something destructive" | **Read this one if you run with `--sandbox local`.** Both halves were wrong. `cwd` is not isolation — a command under the local backend wrote a file to `/tmp`, outside the workspace. And the guard did not stop **`sh -c "rm -rf /"`**: it parses shell syntax, so an interpreter with an inline string was opaque to it. Same for `bash -c`, `python -c "…os.system(…)"`, `perl -e`, `node -e`, `eval`, `find / -delete`. **Now fixed:** the guard recurses into a shell's inline program, refuses inline interpreter code that shells out, and blocks destructive `find` — 18 attacks blocked, 0 false positives. **But the design point stands: a command deny-list can never be complete. The container is the security boundary, not the guard.** The local sandbox is convenience, not a boundary — run untrusted or agent-authored commands with `--sandbox docker` (verified: the same bypass touched nothing on the host under Docker). Note `auto` falls back to local when no image is built — see errata #2. |
| 9 | **Ch 29** — Document Sync, **Ch 31** — Diagnostics | Ch 29 says that once a file is synced, diagnostics are "already sitting there, current as of the last edit" by the time the model checks | They are not. The server analyses in the background and pushes results **~1–1.5 s later**, while the agent writes on one step and calls `get_diagnostics` on the next. The tool read the store immediately, found it empty, and formatted empty as **"No diagnostics reported."** — a confident all-clear for code the server had not looked at yet, in exactly the case the tool exists for. Observed live: the agent wrote `def total(items): return sum(itms)` and was told its code was clean. **This also means errata #5's "verified" note was luck** — the plugin install was genuinely broken and genuinely fixed, but the check that followed passed on timing, not correctness. `_get_diagnostics` now awaits `await_diagnostics()`, and each sync calls `_expect_diagnostics()` to drop the stale result and record that a fresh one is owed (without which a waiter returns the diagnostics from *before* your edit). **If your `get_diagnostics` ever returns nothing for code you know is broken, update — this is why.** |
| 8 | **Ch 26** — Search Tools, and the "exact answer" claim in **Ch 25** | `_find_importers` passes your `name` straight to `importers(name)` | The imports table is keyed by **module** — `from models import User` records `models` — but the model has only ever seen a *directory listing*, so it asks about **`models.py`**. That matched nothing, and the tool replied **"No files import 'models.py'"** about a module three files import. Nothing errored; it simply said *nothing depends on this* about the thing you were about to change — the one direction impact analysis must never fail in. `_find_importers` now normalises the name (strips a leading path and a known source suffix) before querying, so the question the model naturally asks is the one the graph answers. The general point, now in Ch 26: **a tool's contract is not what its docstring says — it's what the model actually passes it.** |
| 7 | **Ch 34** — Hard Truncation | `_hard_truncate` shrinks whichever message is **largest**, and the chapter concludes the agent "never gets silently lobotomized" | Both are wrong, and the second because of the first. The system prompt is **~961 tokens** (rules + tool schema), so once a big tool result has been cut a few times the *system prompt* becomes the largest message — and the loop starts eating the agent's own rules. Measured: with a 10k-token file read, the system prompt was truncated at every `--num-ctx` up to **2560**, even where the budget had room for it. It now spares the system prompt until everything else is exhausted, and warns with the context size you need. **Practical effect:** the rules survive from `--num-ctx` **2304** (the default 8192 was never affected). Related: the ~35-char truncation marker left a floored message at ~235 chars, so the `len(content) <= 200` stop never fired and the loop spun to its 1000-iteration guard on every oversized call; termination is now by progress. |
| 6 | **Ch 39** — Human Escalations, **Ch 10** — Execution Loop | Giving up and the no-progress abort `console.print` + audit + transition | They never **emitted** an event, so every non-terminal surface — the web dashboard and the VS Code extension — showed a bare `error` with **no reason why the run stopped**, while a terminal user watching the same run saw the explanation. Both now emit (`give_up`, `no_progress`) and both UIs render them. Any decision worth printing is a decision worth emitting. |

---

Errata 2, 4, 5 and 6 were each found the same way: by using the agent, not by testing it.
A unit test asserts that a function returns what you expected. It cannot notice that the
dashboard has no input box, that a secret reached the console, that the LSP is quietly
answering "no diagnostics" for every file, or that the agent gave up without telling you
why. Those only surface when you sit down and drive the thing.
